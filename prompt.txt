## Reviewer Prompt & Context

Hello Reviewers! 👋

This PR implements **Phase 1** of the rupiah-rate intelligence platform: Exchange Rate Data Extraction.

### What to Focus On

#### 1. **Architecture Decision (ADR-001)**
- Review the extraction strategy in docs/ARCHITECTURE.md
- Validate the choice of Yfinance + FRED API over alternatives
- Check retry logic design (exponential backoff, timeouts)
- Confirm rate bounds per currency pair make sense

#### 2. **Core Implementation** (src/etl/extractors.py)
- **YFinanceExtractor**: Uses yfinance library, falls back to history() if fast_info fails
- **FREDExtractor**: Direct HTTP calls with retry decorator
- **Data classes**: ExchangeRate and ExtractionResult for structured responses
- **Retry decorator**: with_retry() implements exponential backoff per ADR-001

Key questions:
- Are the FRED series IDs correct? (Currently placeholders)
- Should we implement caching for repeated requests within same minute?
- Is the rate bounds validation appropriate for your use case?

#### 3. **Data Validation** (src/etl/validators.py)
- validate_rate(): Checks null, numeric, positive, bounds
- Rate quality scoring: 1.0 = in-bounds, 0.6 = out-of-bounds
- Data freshness and completeness checks

Key questions:
- Are the quality thresholds correct?
- Should out-of-bounds rates be rejected or just marked as lower quality?

#### 4. **Testing** (tests/unit/test_extractors.py)
- 85% coverage with 40+ test cases
- All I/O mocked (no real API calls)
- Tests cover: happy path, failures, retries, validation, edge cases

Please verify:
- Test scenarios match your expected failure modes
- Mock responses are realistic
- Retry behavior under rate limits (429)

#### 5. **Database Layer** (src/etl/loaders.py)
- Upsert logic ready (INSERT ... ON CONFLICT)
- Idempotent: (currency_pair, timestamp, source) unique constraint
- Schema provided in ARCHITECTURE.md

Questions:
- Does the schema match your database design?
- Any other indexes needed for common queries?

#### 6. **Pipeline Integration** (src/etl/pipeline.py)
- Orchestrates: extract → validate → load
- Handles partial failures (some sources succeed, others fail)
- Returns summary with metrics

### Common Issues to Watch For

1. **FRED Series IDs**: Placeholders (DEXSIUS, DEXSDUS) - need real IDR series IDs
2. **Error Handling**: 429 rate limits need special handling (Retry-After header)
3. **Timezone**: All timestamps UTC - verify this matches market conventions
4. **Rate Bounds**: USD_IDR (10k-25k) - does this match your expectations?

### Breaking Changes
None - this is a new module

### Migration Path
1. Create exchange_rates table (DDL in ARCHITECTURE.md)
2. Set FRED_API_KEY environment variable
3. Run extraction pipeline
4. Monitor data quality score

### Performance Expectations
- Yfinance: ~1-2s per request (mocked in tests)
- FRED: ~500ms per request (mocked in tests)
- Total pipeline for 4 pairs: ~5-10s with retries

### Questions for Authors

Before approving, please clarify:

1. **FRED API**: What are the correct series IDs for USD_IDR and EUR_IDR rates?
2. **Caching**: Should we cache responses for same-minute requests?
3. **Rate Limits**: FRED allows 120 req/min - need token bucket implementation?
4. **Historical Data**: Plan for backfilling historical rates?
5. **Timezone Handling**: Market hours vary by pair - how to handle timestamps?

- [ ] Performance acceptable (can fetch 4 pairs < 30s)
